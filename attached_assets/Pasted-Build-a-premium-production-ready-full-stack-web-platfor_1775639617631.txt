Build a premium, production-ready, full-stack web platform called **Tasty Crousty**.

This project is a **smart food ordering and delivery platform** with a unique operational logic:
**restaurants do NOT begin preparation immediately after an order is placed**.
Preparation only starts **after a driver is assigned and the customer’s delivery details are confirmed**.

The platform must be designed as a serious startup-grade product, not a demo app.

==================================================
CORE PRODUCT POSITIONING
==================================================

Tasty Crousty is not a basic delivery app.

It is a **loss-prevention delivery platform** designed to:
- reduce order cancellations
- reduce fake orders
- reduce restaurant waste
- improve delivery reliability
- validate real customer intent before food preparation begins

Main concept:
1. Customer places an order
2. Restaurant receives the order but preparation is locked
3. System dispatches nearby drivers
4. First driver who accepts is assigned
5. Driver validates the address and order details with the customer
6. Only after successful confirmation does the restaurant start preparing
7. Driver picks up the order
8. Customer tracks delivery in real time
9. Delivery is completed securely with QR code verification

==================================================
PRODUCT WORKFLOW / ORDER LOGIC
==================================================

This is the most important logic in the whole system.

The restaurant must NOT start preparing as soon as the order is placed.

The order must follow this flow:

STEP 1 — CUSTOMER CREATES ORDER
- customer selects city / zone
- customer selects restaurant
- customer adds products and options
- system calculates total amount
- system calculates estimated time
- customer confirms order
- create order in status: pending_dispatch

STEP 2 — RESTAURANT IS NOTIFIED BUT PREPARATION IS LOCKED
- restaurant receives real-time notification
- restaurant sees full order information
- restaurant sees customer address and notes
- restaurant sees a clear visual warning:
  “Waiting for driver confirmation before preparation”
- restaurant cannot start preparing yet

STEP 3 — SMART DRIVER DISPATCH
- system selects drivers based on:
  - proximity to restaurant
  - availability
  - active zone
  - rating
  - average delivery time
  - acceptance rate
  - current workload
- notify only relevant nearby drivers
- first driver who accepts becomes assigned
- assignment is locked for others

STEP 4 — DRIVER PERFORMS CUSTOMER CONFIRMATION
- assigned driver confirms with customer:
  - exact address
  - landmark / building / floor
  - phone accessibility
  - order clarity
- driver chooses one result:
  1. confirmed
  2. needs correction
  3. failed / unreachable customer

IF confirmed:
- order goes to confirmed_for_preparation

IF needs correction:
- order goes to needs_update
- customer or support can update order details

IF failed:
- order goes to confirmation_failed
- restaurant must not prepare

STEP 5 — PREPARATION STARTS ONLY AFTER CONFIRMATION
- once the order is confirmed, restaurant gets a strong real-time approval signal
- restaurant can now start preparing
- order goes to preparing
- when ready, restaurant marks ready_for_pickup

STEP 6 — PICKUP AND DELIVERY
- driver marks picked_up
- status becomes picked_up
- then on_the_way
- then arriving_soon

STEP 7 — REAL-TIME TRACKING
- customer sees:
  - live delivery progress
  - ETA
  - driver name
  - driver status
- platform sends notifications for:
  - order received
  - waiting for driver
  - driver assigned
  - order confirmed
  - order being prepared
  - order ready
  - driver picked up order
  - driver on the way
  - driver arriving soon
  - order delivered

STEP 8 — SECURE DELIVERY WITH QR CODE
- customer has a QR code in order details
- at arrival, driver scans the QR code
- QR scan confirms successful delivery handoff
- order becomes delivered
- payment validation / settlement logic can be triggered after this
